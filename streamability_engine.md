# An algebraic engine for deriving streaming kernels

## Implementation report

This document specifies an engine that, given a dataflow graph of tensor
operators, (1) classifies every axis by the algebraic structure of its
dependence, and (2) for the streamable axes, derives the accumulator type and
combine operator of a single-pass fused kernel. The canonical target it must
reproduce is FlashAttention: given `softmax(QKᵀ)·V`, the engine should tag the
key axis as foldable and emit the `(m, ℓ, o)` online-softmax accumulator with
its rescaling combine, without that accumulator being hand-written anywhere.

The engine is built on one theorem, applied twice.

---

## 1. The core principle

A computation can be streamed (folded in one pass with bounded state) and
parallelised (split into blocks combined in a tree) along an axis **if and only
if the computation, viewed as a function of the list of elements along that
axis, is a monoid homomorphism**:

```
H(xs ++ ys) = H(xs) ⊗ H(ys)
```

where `⊗` is an associative operator with identity `e` on some carrier type.
The inputs combine by concatenation `++`; streaming/tiling is legal exactly when
the outputs combine by a fixed associative `⊗`. Associativity is the single
load-bearing property: it is simultaneously the streaming certificate (fold
left-to-right, O(1) state) and the parallelism certificate (combine blocks in
any grouping, log depth).

Two distinct algebraic facts are used by the engine and must be kept separate:

1. **Monoid structure → fusion/streaming exists.** A reduction over an
   associative `⊕` is a monoid homomorphism; this is what permits one-pass
   evaluation and tiling.

2. **Semiring distributivity → a global normalizer can be deferred.** When the
   consuming reduction is *linear* in its input (a semimodule homomorphism over
   a `(⊕, ⊗)` semiring), a global scalar factors out of the reduction:
   `Σᵢ (c·pᵢ)·vᵢ = c·Σᵢ pᵢ·vᵢ`. This is the property that rescues
   `softmax ∘ reduction`: softmax alone outputs a vector whose normalization is
   entangled with every component (not one-pass streamable), but the attention
   output is a *linear* reduction of that vector, so the normalizer `1/s` slides
   out and is applied once at the end. The engine must model linearity as a
   first-class structure, not just associativity.

The discovery problem the engine automates is **accumulator strengthening**: the
naive output of a function is frequently *not* a homomorphism (the mean of a
concatenation is not a function of the two sub-means), but becomes one once the
carrier is enriched with exactly the state the combine needs (`(sum, count)`).
The engine derives or synthesizes that enriched carrier.

## 2. Architecture overview

The engine is three stages. Stage 1 and Stage 2A are decidable, total, and
compositional — these are the buildable core. Stage 2B is a bounded search
reserved for carriers that are not compositions of known ones.

| Stage | Input → Output | Method | Status |
|-------|----------------|--------|--------|
| 1. Structure classification | DAG → per-(node, axis) structure tag | Abstract interpretation over a finite lattice | Buildable; total |
| 2A. Carrier derivation | streamable axis → (Acc, ⊗, e, project) | Structural recursion over carrier objects | Buildable; reproduces FlashAttention |
| 2B. Carrier synthesis | spec + budget → (Acc, ⊗, e, project) | SMT search modulo associativity | Open; bounded/partial |

Stage 1 is a prerequisite and pruner for Stage 2: it identifies *which* axes are
worth deriving a carrier for and refuses opaque ones. Sequential dependence is
reserved for a future recurrence node with an executable step body; the current
semantic IR does not fabricate it from an operator label.

## 3. Intermediate representation

### 3.1 Primitive operators

The IR is generated by a small basis. Every operator declares, per axis, an
**access function** (the affine map from iteration index to the memory cells it
touches) and whether it writes. The basis:

- **`Input`** — raw data; no dependence on any axis.
- **`Map(f, inputs...)`** — elementwise/broadcast; access function is the
  identity (or a projection for broadcast); introduces no dependence on any axis.
- **`Reduce(src, axis, ⊕)`** — folds `axis` with binary operator `⊕`; the write
  access projects `axis` out.
- **`Scan(src, axis, ⊕)`** — inclusive prefix fold over `axis`; the current IR
  accepts scalar monoids only.
- **`Gather(src, index, axis)`** — `src[index[...]]`; access function is
  **data-dependent** (the index is a tensor, not an affine function).

The first three generate the dense affine core. `MatMul`, `Softmax`,
`LogMatMul`, `LayerNorm`, and attention are **derived** as compositions of
`Map` and `Reduce`, so the semantic IR needs no workload-specific operators.
Product-valued linear-attention/SSM recurrences remain future work:

```
MatMul(A, B; contract=k)  ≜  Reduce( Map(×, A, B), k, ADD )
Softmax(x; axis=k)        ≜  let m = Reduce(x, k, MAX)
                              let e = Map(exp∘sub, x, m)
                              let s = Reduce(e, k, ADD)
                              Map(÷, e, s)
Attention(Q, K, V)        ≜  MatMul( Softmax(MatMul(Q,K;d), k), V; contract=k )
```

### 3.2 Operator algebra metadata

Each binary operator carries its laws, not just a symbol:

```
Operator {
  symbol:        str
  associative:   bool
  commutative:   bool          // affects whether tree-reduce needs order care
  identity:      Option<Value> // None ⇒ no neutral element
  // for Stage 2A, the actual fold semantics:
  carrier:       Type          // the accumulator this operator folds into
  into:          Value -> Acc  // lift one element into the carrier
  combine:       (Acc, Acc) -> Acc
  project:       Acc -> Value
}

is_monoid := associative ∧ identity.is_some()
```

`+`, `×`, `max`, `min`, `logsumexp` are the scalar monoids admitted today. A
future nonlinear recurrence step (e.g. `tanh(W·h + x)`) would not be admitted
through this interface: it needs an explicit executable step and an honest
sequential execution contract. **Soundness obligation:** the declared laws are
trusted by the whole engine; mis-declaring associativity yields incorrect
fusions. The operator library should ship with machine-checked proofs (or
property-tested evidence) of each declared law.

## 4. Stage 1 — structure classification

### 4.1 The structure lattice

Each (node, axis) pair is assigned an element of a finite lattice ordered from
most to least parallel:

```
FREE        no dependence along the axis        → grid / DOALL
MONOIDAL    associative ⊕ with identity          → stream or tree-reduce, O(1) state
LINEAR      a MONOIDAL fold that is additionally  → permits normalizer deferral
            a semimodule homomorphism (linear)      (subtype of MONOIDAL)
OPAQUE      data-dependent access (gather)        → runtime-determined
SEQUENTIAL  dependence with non-associative ⊕     → strict serial, depth = N
```

The executable core currently admits the `FREE`, `MONOIDAL`, and `OPAQUE`
subset. `Reduce` and `Scan` accept scalar monoids only. `SEQUENTIAL` remains a
model-level category for a future recurrence node with an explicit step body;
it is not represented by a name-only operator tag.

`LINEAR` refines `MONOIDAL`: every linear reduction is monoidal, but the engine
records linearity separately because Stage 2A needs it to decide normalizer
deferral. The join (used to combine the structures of a node's inputs) takes the
**least-parallel** of its arguments:

```
FREE < MONOIDAL < LINEAR? ... (see note) ... < OPAQUE < SEQUENTIAL
```

Note: `LINEAR` is not strictly below `OPAQUE` on the parallelism axis; it is a
*tag* attached to a `MONOIDAL` classification rather than a distinct rung. Model
the structure as a pair `(parallelism_level, is_linear: bool)` to avoid forcing
a false total order. The join takes the max parallelism_level and ANDs the
linearity flags.

### 4.2 Propagation rules (per axis `a`)

Computed by structural recursion; `up` is the join of the input structures.

- **`Input`** → `FREE`.
- **`Map`** → `up` (elementwise introduces no dependence; passes structure
  through unchanged). Linearity flag: a `Map` preserves linearity iff `f` is
  linear in the relevant argument (e.g. scaling, addition preserve it; `exp`,
  `gelu` do not — but note these sit *between* reductions, see §5.3).
- **`Reduce(_, axis, ⊕)`**:
  - if `axis ≠ a` → `up` (reducing a different axis is per-`a` independent).
  - if `up` is `OPAQUE` → propagate it (poisoned upstream).
  - else → `MONOIDAL`; set `is_linear` iff `⊕` is `+` (or another
    semiring-additive operator) **and** the summand is linear in the reduced
    quantity.
- **`Scan(_, axis, ⊕)`**:
  - if `axis = a` → `MONOIDAL` (only scalar monoids are admitted).
  - else → `up`.
- **`Gather(_, index, axis)`**:
  - if `axis = a` → `OPAQUE` (join with `up`).
  - else → `up`.

### 4.3 Output

For each output (node, axis): the structure tag. The verdict
`streamable(node, a) := parallelism_level(node, a) ∈ {FREE, MONOIDAL}`.

**Critical refinement over a naive implementation:** track structure
**per-(node, axis)**, not collapsed to the output. The same axis can be reduced
in one sub-expression and free in another (in double-GEMM, the middle axis `k`
is the *contraction* of GEMM-2 but a *free output index* of GEMM-1). The
per-node map is what distinguishes the two fusion kinds in §5.

## 5. Stage 2A — carrier derivation by composition

This is the heart of the engine: turning a `MONOIDAL`/`LINEAR` tag into a
concrete accumulator. The accumulator is derived, not searched, whenever the
computation is built from operators whose carriers are known and whose
composition is governed by the rules below.

### 5.1 The target object

For a streamable axis the engine emits an implementation of:

```
Streamable {
  Item                       // one element along the axis
  Acc                        // the carrier (the derived accumulator type)
  Out                        // the final answer
  into:    Item -> Acc
  combine: (Acc, Acc) -> Acc // associative ⊗
  identity: Acc              // e
  project: Acc -> Out        // applied once, at the end
}
```

`Acc = Out` exactly when the function is homomorphic on its naive output (no
strengthening needed, e.g. `sum`). When `Acc ≠ Out`, the gap is the strengthening
and `project` discards the surplus.

### 5.2 Composition rules

The carrier of a composite expression is built from its parts:

**(R1) Single reduction.** `Reduce(_, a, ⊕)` over a monoid `⊕` contributes
`Acc = ⊕.carrier`, `combine = ⊕.combine`, `into = ⊕.into`, `identity = ⊕.identity`,
`project = ⊕.project`.

**(R2) Map fused into a producer reduction.** `Reduce(⊕) ∘ Map(f)` on the same
axis: push `f` into `into` (`into' = ⊕.into ∘ f`). Carrier unchanged. A `Map`
*after* the reduction composes into `project` (`project' = g ∘ project`).

**(R3) Parallel reductions over the same axis → product carrier.** Two
reductions of the *same* list over the same axis (e.g. `max` and `sum` in
softmax) fuse into the **product monoid**: `Acc = (Acc₁, Acc₂)`,
`combine = (⊗₁, ⊗₂)` componentwise — *unless* the two are coupled (next rule).

**(R4) Coupled reductions (rescaling) → strengthened product.** When a later
reduction's `into`/`combine` depends on a quantity computed by an earlier
reduction over the same axis (softmax's sum is referenced to the running max),
the product carrier acquires a **rescaling combine**. The canonical instance:
`max`-monoid coupled with a `Σexp(·−max)` sum yields

```
Acc = (m, s)
combine((m₁,s₁),(m₂,s₂)) = let M = max(m₁,m₂)
                            (M, s₁·exp(m₁−M) + s₂·exp(m₂−M))
identity = (−∞, 0)
```

This is the online-softmax monoid. Associativity holds because `max` is
associative and the exponential rescales telescope
(`exp(a−b)·exp(b−c) = exp(a−c)`). The engine recognizes this pattern from the
operator metadata (a `MAX` reduction feeding the shift of an `exp` inside an
`ADD` reduction) and emits the coupled combine. **This rule is the encoded form
of the "rescale on new max" insight; it is the part of FlashAttention that is
mechanical once expressed as a composition rule.**

**(R5) Linear consumer → defer the normalizer (extend the carrier).** When a
`LINEAR` reduction (a `+`-reduction, semimodule homomorphism) consumes the
output of a normalized reduction, distributivity lets the normalizer factor out.
The engine extends the carrier with the *unnormalized* accumulator and moves the
division into `project`:

```
softmax(x)·v summed  →  Acc = (m, s, o)
into(x, v)            = (x, 1, v)            // o starts unnormalized
combine              = rescale m, s, AND o together by exp(m_old − m_new)
project((m, s, o))   = o / s                // single deferred division
```

This is the `(m, ℓ, o)` FlashAttention accumulator. It is produced by applying
(R4) to the softmax sub-expression and then (R5) to extend the carrier through
the linear V-reduction. Both steps are rule applications, not search.

### 5.3 Worked derivation: FlashAttention

```
Attention(Q,K,V) = MatMul( Softmax( MatMul(Q,K; d), k ), V; contract=k )
```

1. Inner `MatMul(Q,K; d)` reduces axis `d`, not `k`. Over `k` its output (the
   scores) is `FREE`. Carrier work for `k` starts at the softmax.
2. `Softmax(_, k)` over `k`: `Reduce(MAX,k)` and `Reduce(ADD,k)` coupled through
   the `exp(x−m)` shift → rule (R4) → carrier `(m, s)`, online-softmax combine.
3. Outer `MatMul(_, V; contract=k)` is a `+`-reduction over `k`, i.e. `LINEAR`.
   It consumes the normalized softmax weights → rule (R5) → extend to `(m, s, o)`,
   move `÷s` into `project`.
4. Emit the `Streamable` impl with `Acc = (m, s, o)`, the joint rescaling
   `combine`, `identity = (−∞, 0, 0)`, `project = o/s`.

The `exp`, `gelu`, etc. `Map`s between the reductions do not obstruct anything:
by (R2) they ride inside `into`/`project` and contribute `FREE` to the
structure. Note this is *not* a counterexample to "nonlinearity blocks fusion":
the RNN's nonlinearity sits *inside a same-axis recurrence* (destroying
associativity, §6); softmax's `exp` sits *between two reductions* applied
pointwise, which (R2) absorbs.

### 5.4 Other instances the same rules produce

- **mean** → `(sum, count)`, `project = sum/count` (strengthening via R3/product).
- **variance** → `(count, mean, M2)` with the Welford/Chan parallel combine.
- **logsumexp / log-space matmul (HMM-CRF forward)** → `(max, Σexp(·−max))`,
  `project = log(s)+m` (R4). Naive form materializes the full `[i,j,k]`
  log-score tensor; the rule collapses it to O(1) state over the contracted axis.
- **layernorm/rmsnorm** → mean/variance carriers fused with the elementwise
  affine epilogue in `project`.
- **linear attention / SSM scan** `h_t = A_t·h_{t−1} + b_t` would use affine-map
  composition and the carrier `(A, b)`, once product values give that statement
  executable IR semantics.

## 6. The boundary: what Stage 2A cannot do

Two failure modes, both correctly flagged by Stage 1 rather than mis-derived.

**Non-associative recurrence (SEQUENTIAL).** `h_t = tanh(W·h_{t−1} + x_t)`. The
step does not close under composition — `tanh(W·tanh(W·h+x₁)+x₂)` is not of the
form `tanh(W·h+x′)` — so no bounded `Acc` admits an associative `combine`. The
`Streamable` impl is unimplementable; the axis is a strict serial chain of depth
`T`. This is the genuine sequential atom; it is also the structure of the
autoregressive decode loop at whole-model scale.

**Data-dependent access (OPAQUE).** Embedding lookup, KV-cache indexing, MoE
dispatch, and the gather half of sampling have access functions that are runtime
data, not affine maps. They are neither foldable nor strictly sequential but
*runtime-determined*; the scheduler must handle them with dynamic mechanisms
(paged caches, sorted dispatch). Refinement worth implementing: split `OPAQUE`
by index structure (permutation / sorted / arbitrary) to distinguish coalesced,
schedulable gathers from scattered, bandwidth-bound ones.

The clean statement of the boundary: the primitives capture **dense linear
algebra with static affine access and associative reductions** — most of the
*compute* in an LLM. They do not capture the **routing/control layer**
(embeddings, sampling, MoE, the decode loop), which is exactly the OPAQUE +
SEQUENTIAL residue.

## 7. Stage 2B — carrier synthesis (research frontier)

When the required carrier is not a composition of known carriers — i.e. a
genuine strengthening nobody encoded — Stage 2A has no rule to fire and the
engine must *invent* the accumulator. This is auxiliary-variable synthesis and
is undecidable in general; it becomes decidable under a state-size budget.

**Formulation.** Fix a carrier shape (a tuple of ≤ k scalars). Ask an SMT solver
for the existence of `into`, `combine`, `identity` built from the allowed
operator set such that:

1. `combine` is **associative** (one universally-quantified equation), and
2. `project ∘ fold = spec` (equivalence to the reference semantics).

Search `k = 1, 2, 3, …` upward; the first satisfiable `k` yields the minimal
carrier. This re-derives `(max, sum)` as the `k = 2` solution for softmax
without it being supplied. The cost is exponential in `k` and in the operator
vocabulary; this is the "third homomorphism theorem" automation line.

**What synthesis cannot supply: the specification reframing.** The
FlashAttention leap required noticing that the *target* is `O = A·V`, not `A` —
softmax alone provably has no one-pass form, and only the attention-specific
fact that the consumer is linear makes deferral possible. A synthesizer given
the spec `softmax` will correctly fail; given the spec `softmax then linear
reduction` it can succeed. Choosing the right specification boundary (fusing
across the matmul rather than treating softmax as a sub-result) is a
vocabulary-extending act that lives above term rewriting. Equality saturation
can *re-derive* both online softmax (a reassociation within fixed vocabulary)
and FlashAttention (if the deferred-normalization-with-surrogate rule is in the
rule set), but cannot *invent* the surrogate from a rule set lacking it.
Synthesis-modulo-associativity with auxiliary-state introduction is the right
tool for that frontier, and making it goal-directed enough to rediscover these
carriers from scratch is open.

## 8. Relationship to the scheduling layer

The engine decides **legality and structure**, not **profitability**. Stage 1+2
output: which axes are grid (FREE), which fold (MONOIDAL/LINEAR), and the
concrete accumulator for the folds. They do *not* decide:

- tile/block sizes (set by SRAM/register capacity, occupancy);
- whether a MONOIDAL axis is **streamed** (sequential inner loop, O(1) state —
  for the long axis you avoid materializing, e.g. attention's `k`) or
  **tree-reduced** (parallel across threads — for a short critical-path axis,
  e.g. a dot-product `d`). Both are licensed by associativity; the choice is
  hardware-driven;
- the two fusion *kinds* — same-axis monoid-merge (FlashAttention: collapse
  reductions sharing `k` into one O(1) fold) vs. cross-axis producer-consumer
  tiling (double-GEMM / fused MLP: keep an O(tile) slab of the intermediate in
  SRAM). Both surface as all-FREE+MONOIDAL in Stage 1; the per-(node,axis) map
  from §4.3 distinguishes them, but selecting and sizing them is scheduling;
- whether a legal fusion actually beats the unfused baseline on a given shape.

The engine hands the scheduler a constrained, correct-by-construction search
space (often a single legal fused skeleton plus its accumulator); the scheduler
picks the numbers inside it.

## 9. Implementation checklist

1. **IR + operator library** with per-axis affine access functions and
   machine-checked algebraic-law metadata (associativity, identity, linearity,
   and the carrier `(into, combine, identity, project)` for each monoid).
2. **Stage 1 analyzer**: per-(node, axis) structure inference over the
   `(parallelism_level, is_linear)` lattice by structural recursion; total and
   terminating. Emit the structure map and the `streamable` predicate.
3. **Stage 2A deriver**: carrier composition rules R1–R5; emit the `Streamable`
   record (Acc type, combine, identity, project) for each streamable axis.
   Validate by property-testing the homomorphism law, associativity, identity,
   and `project∘fold = reference` on random data and random splits before
   trusting a derived carrier.
4. **Stage 2B synthesizer** (optional/advanced): SMT search modulo associativity
   over bounded carrier shapes, gated on Stage 1 marking the axis streamable and
   Stage 2A finding no compositional rule.
5. **Scheduling interface**: expose the structure map and accumulators to a
   downstream tiling/codegen pass; do not bake tile sizes or stream-vs-tree
   choices into the algebraic stages.

## 10. Acceptance tests

The engine is correct on the core if, with no hand-written accumulators, it:

- tags matmul `i,j` FREE and `k` MONOIDAL+LINEAR;
- tags attention `s_q` FREE, `k` MONOIDAL, and derives `Acc = (m, s, o)` with
  the online rescaling combine and `project = o/s`;
- derives `(sum,count)` for mean, `(count,mean,M2)` for variance,
  `(max,Σexp)` for logsumexp;
- evaluates and emits scalar-monoid prefix scans, with Add-scan autodiff;
- refuses to represent product-valued or non-associative recurrences until
  their value and step semantics exist;
- tags embedding/sampling/MoE-dispatch axes OPAQUE.

Reproducing the FlashAttention accumulator from the composition rules — not from
a stored template — is the primary success criterion.
